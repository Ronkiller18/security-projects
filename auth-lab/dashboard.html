🔴 Broken Access Control

Critical

Description:
The dashboard page does not verify whether the user is authenticated before granting access.

Vulnerability:
Users can directly access this page by entering the URL, bypassing the login process.

Impact:
Attackers can access restricted areas without authentication, potentially exposing sensitive data or functionality.

Example Attack: 


            Open directly:
            /dashboard.html

Fix:
Validate authentication before allowing access to protected resources. This is typically enforced on the server.

🧠 Vulnerability Breakdown

Educational
This lab intentionally demonstrates insecure authentication patterns commonly found in poorly designed frontend applications.

🔴 Weak Authentication

Login logic is fully client-side and can be modified. 


            login = function() {
                document.getElementById("message").textContent =
                "Login successful";
            }

🟠 Insecure Session

Session stored in localStorage can be manipulated. 


            localStorage.setItem(
                "isLoggedIn",
                "true"
            );

🟠 Session Fixation

A session fixation vulnerability occurs when an application accepts a session identifier supplied by an attacker instead of generating a new one after authentication. 


            localStorage.setItem(
                "sessionId",
                "attacker-session-123"
            );

If the victim logs in using this fixed session, the attacker already knows the authenticated session value.

🔴 Broken Route Protection

Protected routes should verify authentication before granting access. 

/dashboard.html

🟠 Privilege Escalation

The application trusts roles stored in localStorage. Attackers can modify their own privileges. 


            localStorage.setItem(
                "userRole",
                "admin"
            );

This allows unauthorized administrative access.

🟢 Secure Mitigations

  • Regenerate sessions after login
  • Invalidate sessions during logout
  • Enforce authentication checks on protected routes
  • Validate authorization server-side
  • Never trust localStorage for security decisions

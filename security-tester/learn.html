Project Overview

Client-Side Security Tester is a browser-based educational security analysis tool designed to demonstrate common client-side security risks. The project performs heuristic analysis against user-supplied input and highlights potentially dangerous patterns commonly associated with web application attacks.

All findings are heuristic and intended for learning purposes only. They should not replace a professional security assessment.

Supported Analyzers

  • XSS Detection — script injection patterns and event handlers
  • DOM Sink Analysis — dangerous client-side APIs
  • URL Reputation — phishing patterns and domain analysis
  • JWT Inspection — token structure and security claims
  • Content Security Policy Review — directive strength
  • Security Header Analysis — HTTP response header coverage
  • Cookie Security Analysis — Set-Cookie attribute checks
  • Storage Security Analysis — localStorage and sessionStorage usage
01

Cross-Site Scripting XSS

High Severity +40 pts per finding

Description

Cross-Site Scripting occurs when untrusted input is interpreted as executable JavaScript by a browser. Attackers inject malicious scripts that run in the context of a victim's session.

Why It Matters

Successful XSS attacks can lead to session theft, account compromise, credential harvesting, and malicious actions performed on behalf of users without their knowledge.

Example Payloads

HTML / JS 
<script>alert(1)</script>
<img src=x onerror=alert(1)>
<svg onload=alert(1)>
<body onload=alert(document.cookie)>
<iframe src=x>
javascript:alert(1)

Mitigation

  • Validate and sanitize all user input server-side
  • Escape HTML output before rendering
  • Implement a strict Content Security Policy
  • Avoid rendering raw user-controlled content in the DOM
02

DOM-Based XSS DOM

High Severity +40 pts per finding

Description

DOM XSS occurs when JavaScript reads attacker-controlled input and inserts it directly into dangerous browser APIs — entirely client-side, with no server interaction required.

Why It Matters

Vulnerable DOM sinks may allow attackers to execute arbitrary JavaScript without interacting with a backend server, bypassing many traditional server-side protections.

Dangerous Sinks Detected

JavaScript 
element.innerHTML = userInput
document.write(userInput)
eval(userInput)
element.insertAdjacentHTML('beforeend', userInput)
location.href = userInput
location.assign(userInput)
location.replace(userInput)
setTimeout(userInput, 0)

Mitigation

  • Prefer textContent over innerHTML
  • Never pass user input to eval()
  • Sanitize user-controlled data before DOM insertion
  • Use secure templating systems with auto-escaping

URL Reputation & Phishing URL

High / Medium Severity +40 or +20 pts per finding

Description

The URL Reputation analyzer runs six independent checks against a pasted URL. Each check produces its own finding — a single URL can trigger multiple findings at different severities.

Checks Performed

  • Raw IP as domain — legitimate services never use bare IP addresses as their hostname
  • Excessive subdomains — three or more subdomain levels suggest domain spoofing
  • Brand keyword + suspicious TLD — e.g. paypal-secure.tk combines trust-bait with a free throwaway domain
  • Action keywords + suspicious TLD — login/verify/secure combined with .xyz, .tk, .ml, .ga, .cf
  • URL-encoded characters in hostname — percent-encoding used to obscure the real destination
  • Homograph characters — non-ASCII lookalike letters used to impersonate trusted domains

Example Suspicious URLs

URLs 
http://192.168.1.1/login
http://secure.paypal.com.verify.login.xyz/
https://paypal-secure.tk
https://secure-login-update.xyz
http://pay%70al.com/login

Mitigation

  • Verify the registrable domain — the last two parts before the TLD
  • Never trust login links received via email or SMS
  • Check for percent-encoding by decoding the URL fully before visiting
  • Enable multi-factor authentication to limit damage from credential theft

JWT Security Analysis JWT

Medium / High Severity +20 or +40 pts per finding

Description

JSON Web Tokens are commonly used for authentication and authorization in modern web applications. Misconfigured tokens can expose sensitive data or allow signature bypass entirely.

Detected Issues

  • alg: none — signature verification bypass (case-insensitive check)
  • Missing exp claim — tokens never expire
  • Sensitive key names in payload — password, secret, apikey

Example Vulnerable Token

JWT 
eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJ1c2VyIjoiYWRtaW4iLCJwYXNzd29yZCI6InNlY3JldCJ9.
Note: The token above decodes to alg: none with password: "secret" in the payload — both critical issues detectable by the JWT Analyzer.

Mitigation

  • Always require and verify token signatures using RS256 or HS256
  • Include and validate exp claims on every token
  • Never store secrets, passwords, or API keys in token payloads
  • Rotate signing keys regularly

Content Security Policy CSP

High / Medium Severity +40 or +20 pts per finding

Description

CSP is a browser security mechanism used to restrict the sources from which content may be loaded and executed. A weak or missing CSP leaves users exposed to XSS and data injection attacks.

Detected Issues

  • unsafe-inline — allows inline script execution
  • unsafe-eval — allows eval() and similar dynamic execution
  • Wildcard script-src — scripts loadable from any domain
  • Wildcard object-src — unsafe embedded content allowed
  • Missing frame-ancestors — clickjacking risk

Weak Policy

CSP Header 
default-src 'self';
script-src 'unsafe-inline' *;
object-src *;

Strong Policy

CSP Header 
default-src 'none';
script-src 'self' 'nonce-{random}';
style-src 'self';
img-src 'self' data:;
font-src 'self';
connect-src 'self';
frame-ancestors 'none';
base-uri 'self';
form-action 'self';

Mitigation

  • Use nonces or hashes instead of unsafe-inline
  • Never use unsafe-eval in production
  • Avoid wildcard sources for scripts or objects
  • Always define frame-ancestors 'none'

Security Header Analysis HDR

High / Medium / Low +40, +20 or +10 pts per finding

Description

Paste raw HTTP response headers to check for missing or misconfigured security directives. Each header is checked independently — a single response can trigger multiple findings.

Headers Checked

  • Content-Security-Policy — missing entirely (High)
  • Strict-Transport-Security — missing (High), weak max-age under 6 months (Medium), missing includeSubDomains (Low)
  • X-Frame-Options — missing (Medium), invalid value not DENY or SAMEORIGIN (Medium)
  • X-Content-Type-Options — missing or not set to nosniff (Medium)
  • Referrer-Policy — missing (Low), weak value (Low)
  • Permissions-Policy — missing (Low)
  • X-XSS-Protection — misconfigured legacy value (Low)
  • Cache-Control — missing no-store/no-cache on potentially sensitive response (Low)

Example — Missing Headers

HTTP Headers 
X-Frame-Options: ALLOW
Cache-Control: public
X-XSS-Protection: 1

Example — Strong Headers

HTTP Headers 
Content-Security-Policy: default-src 'self'
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()

Mitigation

  • Add all six security headers to every HTTP response
  • Set HSTS max-age to at least 31536000 (one year)
  • Set X-Frame-Options to DENY unless same-origin framing is needed
  • Use Referrer-Policy: strict-origin-when-cross-origin as a safe default

Cookie Security Analysis CKI

High / Medium / Low +40, +20 or +10 pts per finding

Description

Paste one or more Set-Cookie header lines to check each cookie's security attributes. Every cookie is analyzed independently — a single paste of three cookies can produce fifteen or more findings.

Checks Per Cookie

  • HttpOnly — missing means JavaScript can read the cookie via document.cookie (High if sensitive name, Medium otherwise)
  • Secure — missing means the cookie transmits over unencrypted HTTP (High if sensitive name, Medium otherwise)
  • SameSite — missing exposes the cookie to CSRF attacks (Medium)
  • SameSite=None without Secure — browsers reject this combination and it enables CSRF (High)
  • Both flags missing on sensitive cookie — additional High finding when name suggests auth data
  • No expiry — session cookie persists on shared devices (Low)

Sensitive Cookie Names

The analyzer automatically boosts severity for cookies with names matching: session, token, auth, jwt, user, admin, key, secret, pass, cred.

Example — Insecure Cookies

Set-Cookie 
Set-Cookie: session=abc123; Path=/
Set-Cookie: prefs=dark; SameSite=None

Example — Secure Cookies

Set-Cookie 
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict; Max-Age=3600; Path=/
Set-Cookie: prefs=dark; SameSite=Lax; Max-Age=2592000; Path=/

Mitigation

  • Always set HttpOnly and Secure on authentication and session cookies
  • Use SameSite=Strict for authentication cookies
  • Never use SameSite=None without also setting the Secure flag
  • Set explicit Max-Age or Expires on all cookies

Storage Security Analysis STR

High / Medium / Low +40, +20 or +10 pts per finding

Description

Paste JavaScript code that uses localStorage or sessionStorage to detect sensitive data being stored client-side. Both analyzers use identical detection logic — the tool label distinguishes the source.

Checks Performed

  • Sensitive key in setItem — keys like token, auth, jwt, session, password, secret trigger a High finding
  • JWT value pattern — a base64 eyJ... segment detected in the code (High)
  • Bearer token pattern — Bearer eyJ... style value detected (High)
  • JSON.stringify inside setItem — serialised objects may contain hidden sensitive fields (High if key is sensitive, Medium otherwise)
  • Sensitive key in getItem — confirms sensitive data exists in storage (Medium)
  • eval() with storage value — direct code injection vector (High)
  • General usage notice — fallback Low finding when no sensitive patterns detected

Example — Insecure Storage

JavaScript 
localStorage.setItem('token', jwtValue)
localStorage.setItem('user', JSON.stringify(userData))
const auth = localStorage.getItem('authToken')
eval(localStorage.getItem('script'))

Example — Safer Approach

JavaScript 
// Store auth tokens in HttpOnly cookies via the server
// Use localStorage only for non-sensitive UI state
localStorage.setItem('theme', 'dark')
localStorage.setItem('language', 'en')
Key rule: Never store authentication tokens, JWTs, passwords, or session identifiers in localStorage or sessionStorage. An XSS attack on any script on the page can read all storage instantly. Use HttpOnly cookies instead.

Mitigation

  • Store auth tokens in HttpOnly cookies managed server-side
  • Use localStorage only for non-sensitive UI state (theme, language)
  • Never pass storage values to eval() or Function()
  • Audit all JSON.stringify calls to check what fields are being serialised

Risk Scoring Model

The dashboard generates a cumulative risk score based on the severity of all active findings. Scores are additive — running multiple tools in a session accumulates across findings until the dashboard is cleared.

Severity Points Examples
High +40 XSS, DOM Sinks, Missing HSTS, Unsigned JWT, Missing HttpOnly on session cookie
Medium +20 Phishing URL, Weak CSP, JWT missing exp, Missing X-Frame-Options, Missing SameSite
Low +10 Missing Referrer-Policy, Weak HSTS duration, Cookie without expiry, General storage usage
Disclaimer: This scoring model is intentionally simplified for educational purposes and should not replace professional risk assessment methodologies or formal security audits.
